# ADR-002: 选择 SQLite 作为开发数据库

## 状态
已接受

## 背景

find_job 是一个简历管理系统，数据量预期为：
- 单用户简历数量：1-10 份
- 每份简历模块数：3-10 个
- 每份简历总字段数：< 200 个

数据规模较小，但需要：
- 零配置、零运维的本地开发体验
- 支持 Docker 一键部署
- 便于在不同开发者机器间一致运行

## 决策

开发/测试环境使用 **SQLite**，通过 SQLAlchemy ORM 抽象数据访问。

### 主要理由

1. **零配置**：无需安装数据库服务，文件即数据库
2. **零运维**：不适合生产高并发场景，但完全满足开发需求
3. **版本控制友好**：`.db` 文件可通过 .gitignore 排除，不影响代码仓库
4. **SQLAlchemy 兼容**：通过 ORM 抽象，未来可无缝迁移至 PostgreSQL

### 备选方案对比

| 数据库 | 优势 | 劣势 |
|--------|------|------|
| SQLite | 零配置、零运维、轻量 | 并发写入受限、无网络协议 |
| PostgreSQL | 功能完整、高并发、生产级 | 需要 Docker 服务、更多资源 |
| MySQL | 广泛使用、生态成熟 | 相对重量级 |

## 后果

### 正面
- 开发者克隆仓库后无需安装数据库服务，`pip install` 后即可运行
- Docker Compose 中无需声明额外数据库服务
- 数据存储在单个 `.db` 文件中，备份/迁移简单
- SQLAlchemy 抽象层保证未来可迁移至 PostgreSQL 而无需修改业务代码

### 负面
- SQLite 的 WAL 模式在高并发写入时存在锁竞争（当前场景不构成问题）
- 生产环境如需多实例部署，SQLite 不适用，需迁移至 PostgreSQL
- 未来如引入 AI 特性产生大量日志数据，需要评估数据增长策略

### 迁移计划

当系统需要生产部署时：
1. 切换 `DATABASE_URL` 环境变量至 PostgreSQL 连接字符串
2. 使用 Alembic 执行数据库迁移
3. 无需修改任何业务代码（通过 SQLAlchemy 抽象屏蔽差异）
